## 2. 核心抽象：Outbound Trait

所有代理协议实现同一个 trait，使上层代码与协议实现完全解耦：

```rust
// crates/xtune-core/src/proxy/connector.rs
pub trait Outbound: Send + Sync + 'static {
    fn connect(
        &self,
        host: &str,
        port: u16,
    ) -> Pin<Box<dyn Future<Output = Result<BoxProxyStream>> + Send + '_>>;

    fn name(&self) -> &str;
}

pub type BoxProxyStream = Box<dyn AsyncRead + AsyncWrite + Unpin + Send>;
pub type SharedOutbound = Arc<dyn Outbound>;
```

### RetryOutbound 退避策略

```

```

## 3. 工厂模式：create_outbound

```rust
// crates/xtune-core/src/proxy/factory.rs
pub fn create_outbound(node: &Node) -> Result<SharedOutbound> {
    match &node.protocol {
        // TCP-based：外层 RetryOutbound(3次)
        ProxyProtocol::VLess { .. }   => return Ok(VlessOutbound::new(node)?.with_retry(3)),
        ProxyProtocol::VMess { .. }   => return Ok(VMessOutbound::new(node)?.with_retry(3)),
        ProxyProtocol::Trojan { .. }  => return Ok(TrojanOutbound::new(node)?.with_retry(3)),
        ProxyProtocol::Shadowsocks { .. } => return Ok(SsOutbound::new(node)?.with_retry(3)),

        // QUIC-based：不包 RetryOutbound（内部自管理重连）
        ProxyProtocol::Tuic { .. }      => return Ok(TuicOutbound::new(node)?),
        ProxyProtocol::Hysteria2 { .. } => return Ok(Hysteria2Outbound::new(node)?),
    }
}
```

### 设计理由

QUIC 协议不使用外层 Retry 的原因：

```mermaid
sequenceDiagram

    participant RetryOutbound
    participant TuicOutbound
    participant Server

    Note over Client,Server: ❌ 有外层 Retry 的情况（已移除）
    Client->>RetryOutbound: connect()
    RetryOutbound->>TuicOutbound: attempt 1 connect()
    TuicOutbound->>Server: QUIC 握手 (7s timeout × 2 addrs)
    Server-->>TuicOutbound: 超时
    TuicOutbound-->>RetryOutbound: Err
    RetryOutbound->>TuicOutbound: attempt 2 connect() [+200ms]

    Server-->>TuicOutbound: 超时
    TuicOutbound-->>RetryOutbound: Err
    RetryOutbound->>TuicOutbound: attempt 3 connect() [+2s]
    Note over RetryOutbound: 总等待: 14s + 14s + 14s = 42s ❌

    Note over Client,Server: ✅ 无外层 Retry（当前实现）
    Client->>TuicOutbound: connect()
    TuicOutbound->>Server: 内部重试（2 attempts × 7s per addr）
    Note over TuicOutbound: 总等待: max 14s ✅
```

## 4. 连接生命周期：TCP 协议 vs QUIC 协议

```mermaid
flowchart LR
    subgraph TCP["TCP-based (VLess/VMess/Trojan/SS)"]
        direction TB
        T1["每次 connect() 调用"] --> T2["ConnPool.get()"]
        T2 --> T3{"有缓存连接?"}
        T3 -->|"有，age < 30s"| T4["复用连接"]
        T3 -->|"无或过期"| T5["新建 TCP+TLS 连接"]
        T5 --> T6["写入协议头（握手）"]
        T4 --> T6
        T6 --> T7["返回 BoxProxyStream"]
    end

    subgraph QUIC["QUIC-based (TUIC/Hysteria2)"]
        direction TB
        Q1["每次 connect() 调用"] --> Q2["get_connection()"]
        Q2 --> Q3{"QUIC 连接存活?"}
        Q3 -->|"存活"| Q4["open_bi() 开新 QUIC 流"]
        Q3 -->|"连接已关闭"| Q5["create_connection()"]
        Q5 --> Q6["QUIC 握手 + 协议认证"]
        Q6 --> Q4
        Q4 --> Q7{"open_bi 成功?"}
        Q7 -->|"是"| Q8["返回 BoxProxyStream"]
        Q7 -->|"否（stale race）"| Q9["reconnect + retry"]
        Q9 --> Q4
    end
```

**关键差异**：
- TCP：每个 `connect()` 可能新建 TCP+TLS 连接（约 1-2 RTT 开销）；ConnPool 预建连接缓解此问题
- QUIC：一个 UDP 连接承载所有流；`connect()` 仅需 `open_bi()`（约 0 RTT 开销）；连接关闭后自动重连

## 5. TUIC v5 协议实现

文件：`crates/xtune-core/src/proxy/tuic.rs`

### 5.1 结构体设计

```rust
pub struct TuicOutbound {
    server: String,
    port: u16,
    uuid: [u8; 16],
    password: String,
    sni: String,
    congestion: CongestionControl,          // Cubic | NewReno | Bbr
    quic_client_config: quinn::ClientConfig, // 预构建，跨重连复用
    state: tokio::sync::RwLock<           // 双检锁保护的连接状态
        Option<(quinn::Endpoint, quinn::Connection)>
    >,
}
```

**关键设计**：`quic_client_config` 在 `new()` 时一次性构建（包含 TLS 根证书、传输参数），后续所有重连直接 clone，避免重复构建开销。

### 5.2 常量

```rust
const TUIC_CONNECT_TIMEOUT: Duration = Duration::from_secs(7);
const TUIC_CONNECT_ATTEMPTS_PER_ADDR: usize = 2;
const TUIC_CONNECT_RETRY_BACKOFF: Duration = Duration::from_millis(250);
```

### 5.3 连接管理（双检锁）

```rust
async fn get_connection(&self) -> Result<(Endpoint, Connection)> {
    // 快速路径：读锁
    {
        let guard = self.state.read().await;
        if let Some((ep, conn)) = guard.as_ref() {
            if conn.close_reason().is_none() {  // 连接仍存活
                return Ok((ep.clone(), conn.clone()));
            }
        }
    }
    // 慢速路径：写锁 + 双检
    let mut guard = self.state.write().await;
    if let Some((ep, conn)) = guard.as_ref() {
        if conn.close_reason().is_none() {
            return Ok((ep.clone(), conn.clone()));
        }
    }
    // 创建新连接
    let (ep, conn) = self.create_connection().await?;
    self.authenticate(&conn).await?;
    *guard = Some((ep.clone(), conn.clone()));
    Ok((ep, conn))
}
```

### 5.6 stale 连接竞态修复

```rust
// connect() 中的 open_bi() 重试循环
async fn connect(&self, host: &str, port: u16) -> Result<BoxProxyStream> {
    for attempt in 1..=2 {
        let (_, conn) = self.get_connection().await?;
        match conn.open_bi().await {
            Ok((send, recv)) => {
                // 发送 CONNECT 命令...
                return Ok(Box::new(TuicStream { send, recv }));
            }
            Err(e) => {
                if attempt == 2 { return Err(e.into()); }
                // 连接在 get_connection() 和 open_bi() 之间死亡
                // 下次 get_connection() 会检测 close_reason() 并重连
                tracing::warn!("TUIC open_bi failed (attempt {}): {}", attempt, e);
            }
        }
    }
    unreachable!()
}
```

**竞态窗口**：`get_connection()` 返回一个看起来存活的连接，但在调用 `open_bi()` 之前的毫秒级间隙中，QUIC 连接因 idle timeout 或服务器重启而关闭。`close_reason()` 是在关闭后设置的，所以第一次调用 `get_connection()` 无法检测到。第二次 attempt 触发 `get_connection()` 时，`close_reason().is_some()` 为 true，从而重连。

### 7.1 通用连接模式

```mermaid
    E --> F{"有 TLS?"}
    F -->|"是"| G["TLS 握手（craftls 指纹）"]
    F -->|"否"| H["原始 TCP stream"]
    G --> H
    D --> I["写协议头（目标地址/认证）"]
    H --> I
    I --> J["返回 BoxProxyStream"]
```

### 7.2 ConnPool 与 ConnFactory

```rust
// pool.rs
pub trait ConnFactory: Send + Sync + 'static {
    fn make(&self) -> Pin<Box<dyn Future<Output = Result<BoxProxyStream>> + Send + '_>>;
}

pub struct ConnPool {
    factory: Arc<dyn ConnFactory>,
    capacity: usize,          // 默认 4
    max_age: Duration,        // 默认 30s
    connections: Mutex<VecDeque<(Instant, BoxProxyStream)>>,
}
```

**预热逻辑**：首次 `pool.get()` 调用会在后台触发 `schedule_fill()`，在 50ms 后开始填充连接，避免首次连接延迟与惊群效应。

### 7.3 VMess AEAD 帧格式（简化）

```
客户端 → 服务器:
  [16B IV][16B header][payload chunks...]

  header = AES-128-CFB-encrypt({
    ver=1, IV, key, response_token, options,
    command=TCP, port, addr_type, addr,
    ...
  })

  payload chunks = [{length(2B) + AEAD-tag(16B)}{data}]
```

### 7.4 VLESS 帧格式（简化）

```
客户端 → 服务器:
  [1B ver=0][16B UUID][M addons][1B N][N addon_bytes]
  [1B cmd=TCP][2B port][1B addr_type][addr]
  [payload...]
```

## 8. 传输层：TLS 指纹伪装（craftls）

文件：`crates/xtune-core/src/proxy/transport.rs`，`crates/craftls/`

### 8.1 craftls 是什么

标准 rustls 生成的 TLS ClientHello 有固定的 extension 顺序和 cipher suite 列表，深度包检测（DPI）设备可以识别并阻断。`craftls` 是一个 [3andne/craftls](https://github.com/3andne/craftls) 的分支，允许精确控制 ClientHello 指纹以模拟真实浏览器行为。

### 8.2 支持的指纹

| 指纹名 | 模拟目标 | 用途 |
|--------|---------|------|
| `CHROME_112` | Chrome 112（默认） | 最广泛兼容 |
| `CHROME_108` | Chrome 108 | 备选 Chrome |
| `SAFARI_17_1` | Safari 17.1 | iOS/macOS 场景 |
| `FIREFOX_105` | Firefox 105 | 备选 |

### 8.3 使用方式

```rust
// transport.rs
pub fn build_tls_connector(tls_config: &TlsConfig) -> Result<TlsConnector> {
    let fp_name = tls_config.fingerprint.as_deref().unwrap_or("chrome");
    let fp = fingerprint_builder(Some(fp_name))
        .do_not_override_alpn();  // 保留应用层设置的 ALPN

    let rustls_config = RustlsClientConfig::builder()
        .with_root_certificates(get_root_certs())
        .with_no_client_auth()
        .with_fingerprint(fp);

    Ok(TlsConnector::from(Arc::new(rustls_config)))
}
```

### 8.4 双 TLS 栈架构

xtune-core 内**同时存在两套 TLS 实现**，通过 API 版本隔离，互不干扰：

```mermaid
graph TD
    subgraph TCP["TCP 协议路径 (VMess/VLess/Trojan/SS)"]
        craftls["craftls (rustls 0.22/0.25 fork)\n指纹伪装 ClientHello\nbuild_tls_connector()"]
        craft_tokio["craft-tls (tokio-rustls 0.25)\nTLS over TCP stream"]
        craftls --> craft_tokio
    end

    subgraph QUIC["QUIC 协议路径 (TUIC/Hysteria2)"]
        std_rustls["标准 rustls 0.23\n无指纹伪装（QUIC 协议本身抗 DPI）\nget_quic_root_certs()"]
        quinn_lib["quinn 0.11\nQUIC 传输层"]
        std_rustls --> quinn_lib
    end

    patch["Cargo.toml [patch.crates-io]\ncraftls = { path = 'crates/craftls' }"]
    patch -.->|"替换"| craftls
```

**为何 QUIC 不用 craftls**：QUIC 基于 UDP 且所有数据加密，流量特征与 TLS-over-TCP 不同，DPI 设备识别方式不同。而且 `quinn` 绑定标准 `rustls 0.23` API，craftls 的 0.22/0.25 API 不兼容。

### 8.5 根证书缓存

```rust
static QUIC_ROOT_CERTS: OnceLock<RootCertStore> = OnceLock::new();

fn get_quic_root_certs() -> &'static RootCertStore {
    QUIC_ROOT_CERTS.get_or_init(|| {
        let mut store = RootCertStore::empty();
        store.extend(webpki_roots::TLS_SERVER_ROOTS.iter().cloned());
        store
    })
}
```

根证书在进程生命周期内只构建一次（约 1-2ms），后续所有连接直接引用静态引用。**注意**：craftls（TCP 路径）和标准 rustls（QUIC 路径）各有独立的根证书缓存 static。

## 9. 连接池设计（ConnPool）

```mermaid
stateDiagram-v2
    [*] --> Empty: 初始化（lazy）

    Empty --> Filling: 首次 get() 触发 schedule_fill()
    Filling --> Ready: fill() 成功（最多 capacity 个连接）

    Ready --> Hit: get() 有缓存且 age < 30s
    Ready --> Miss: get() 无缓存或全部过期

    Miss --> Filling: 触发后台 refill
    Miss --> DirectConnect: 直接新建连接（不等待 refill）

    Hit --> Ready
    DirectConnect --> Ready

    Ready --> Evicting: 连接 age >= 30s
    Evicting --> Ready: 过期连接被驱逐，触发 refill
```

**关键参数**：
- `capacity = 4`：预建连接数
- `max_age = 30s`：连接最大存活时间
- `refill_delay = 50ms`：每次 fill 尝试的间隔，避免惊群

**注意**：ConnPool 仅用于 TCP-based 协议。QUIC 协议（TUIC/Hysteria2）通过 `RwLock<Option<Connection>>` 管理单一持久 QUIC 连接。

### 10.1 Relay Future 内部结构

```mermaid
classDiagram
    class Relay {
        a: &mut A
        b: &mut B
        a_buf: CopyBuf   %% A→B 方向的 64KB 缓冲区
        b_buf: CopyBuf   %% B→A 方向的 64KB 缓冲区
        a_to_b: u64      %% 累计 A→B 字节（仅 Ready 时累积）
        b_to_a: u64      %% 累计 B→A 字节（仅 Ready 时累积）
        a_done: bool     %% A 读端已 EOF
        b_done: bool     %% B 读端已 EOF
    }
    class CopyBuf {
        buf: Box[u8;65536]
        pos: usize
        cap: usize
    }
    Relay --> CopyBuf : a_buf
    Relay --> CopyBuf : b_buf
```

### 10.2 transfer_one 状态机

`transfer_one(cx, reader, writer, buf, done)` 在单次 `poll()` 调用中尽可能传输数据：

```mermaid
flowchart TD
    Start([每次 Relay::poll 调用]) --> Check1{"buf.pos < buf.cap?\n（缓冲区有待写数据）"}
    Check1 -->|"是"| Write["poll_write(writer, buf[pos..cap])"]
    Write --> W1{"ready?"}
    W1 -->|"Pending"| RetPending["return Poll::Pending\n⚠️ transferred 局部变量丢失"]
    W1 -->|"Ok(n)"| Acc["buf.pos += n\ntransferred += n"]
    Acc --> Check1
    Check1 -->|"否"| CheckDone{"*done?"}
    CheckDone -->|"是"| Flush["poll_flush(writer)"]
    Flush --> F1{"ready?"}
    F1 -->|"Pending"| RetPending
    F1 -->|"Ok"| RetReady["return Poll::Ready(Ok(transferred))"]
    CheckDone -->|"否"| Read["poll_read(reader, buf)"]
    Read --> R1{"ready?"}
    R1 -->|"Pending"| RetPending
    R1 -->|"n=0 EOF"| SetDone["*done=true; poll_shutdown"] --> RetReady
    R1 -->|"n>0"| SetCap["buf.cap=n"] --> Check1
```

> **⚠️ 统计字节 Bug**：`transferred` 是 `transfer_one` 内的局部变量，在每次调用时从 0 开始。当 `poll_write` 返回 `Pending`（写入被内核缓冲区阻塞），即使此次迭代已经写出部分数据（`transferred > 0`），这些字节数也随 `Poll::Pending` 丢失，不会被 `Relay::a_to_b` 累积。对于 MB 级别的传输，最终 `ProxyStats.bytes_sent/bytes_received` 只会反映连接最后一个 `transfer_one` 周期内的字节数（通常 < 64KB），远低于实际传输量。

### 10.3 优雅关闭序列

```mermaid
sequenceDiagram
    participant A as 流 A（客户端）
    participant R as Relay Future
    participant B as 流 B（代理服务器）

    A->>R: poll_read → n=0（EOF）
    R->>B: poll_shutdown（关闭 A→B 写入方向）
    Note over R: a_done = true
    R->>B: 继续读 B→A 方向，直到 B 也 EOF
    B->>R: poll_read → n=0（EOF）
    R->>A: poll_shutdown（关闭 B→A 写入方向）
    Note over R: b_done = true
    R-->>Caller: Poll::Ready(Ok((a_to_b, b_to_a)))
```

**设计要点**：
- 64KB 缓冲区：减少系统调用频率，提升吞吐量（参考 Clash 实现）
- 全双工：两个方向同时监听，不阻塞彼此
- 优雅关闭：收到 EOF 时 `poll_shutdown` 通知对端写半部关闭，等待另一方向完成
- 返回值：`(bytes_a→b, bytes_b→a)` 理论上用于统计，但因上述 Bug 实际值偏低

```rust
fn stop_proxy(&mut self, cx: &mut Context<Self>) {
    // 1. 停止 TUN（如果活跃）
    if self.tun_enabled { self.stop_tun(cx); }

    // 2. 清除系统代理（如果 app 设置的）
    if self.system_proxy_managed_by_app {
        let _ = sysproxy::Sysproxy::set_enabled(false);
    }

    // 3. 发送停止信号（非阻塞）
    if let Some(tx) = self.proxy_stop_tx.take() {
        let _ = tx.send(());
    }

    // 4. 立即更新状态
    self.proxy_running = false;
    self.proxy_status = "Disconnected".to_string();
    self.active_proxy_node = None;
    cx.notify();
}
```

**关键**：`stop_proxy` 不 await 任何 future，立即返回。旧版实现会等待连通性验证超时（最高 24s），现在通过 `session_id` 机制让验证任务自行终止。

**线程安全保证**：

| 组件 | 线程 | 访问方式 |
|------|------|---------|
| `AppState` 所有字段 | 仅 GPUI 主线程 | `cx.update()` 序列化 |
| `ProxyStats` | 任意线程 | `AtomicU64` |
| `TuicOutbound.state` | 任意 tokio 线程 | `tokio::RwLock` |
| `ConnPool.connections` | 任意 tokio 线程 | `tokio::Mutex` |
| `DnsResolver.cache` | 任意 tokio 线程 | `tokio::Mutex` |
| `Router.cache` | 任意 tokio 线程 | `tokio::RwLock` |

**tokio workers 固定为 2**（`main.rs` 硬编码）：对代理场景（I/O 密集）够用；纯 CPU 密集型任务（GeoIP 匹配 + 路由规则遍历）在 2 个 worker 上轻度竞争。

## 23. RetryOutbound 退避时序

```mermaid
sequenceDiagram
    participant Caller
    participant Retry as RetryOutbound (max=3)
    participant Inner as 内层 Outbound (TCP)
    participant Server as 代理服务器

    Caller->>Retry: connect(host, port)

    Note over Retry: attempt 1 (立即)
    Retry->>Inner: connect()
    Inner->>Server: TCP 连接 [超时 10s]
    Server-->>Inner: Err / timeout
    Inner-->>Retry: Err

    Note over Retry: sleep 200ms (base_delay * 2^0)
    Retry->>Retry: sleep(200ms)

    Note over Retry: attempt 2
    Retry->>Inner: connect()
    Inner->>Server: TCP 连接 [超时 10s]
    Server-->>Inner: Err / timeout
    Inner-->>Retry: Err

    Note over Retry: sleep min(200ms*2, 2000ms) = 400ms
    Retry->>Retry: sleep(400ms)

    Note over Retry: attempt 3
    Retry->>Inner: connect()
    Inner->>Server: TCP 连接 [超时 10s]
    Server-->>Inner: Ok(stream)
    Inner-->>Retry: Ok(stream)
    Retry-->>Caller: Ok(BoxProxyStream)

    Note over Retry,Caller: 最坏情况: 10s + 200ms + 10s + 400ms + 10s = 30.6s
```

**仅 TCP 协议使用 RetryOutbound**；QUIC 协议（TUIC/Hysteria2）内部管理重连，最坏情况 14s（2 地址 × 2 尝试 × 7s 超时）。
